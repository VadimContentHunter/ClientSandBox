# Техническое задание

## Текущее состояние

На данный момент реализовано:

-   новая архитектура вкладки «Подключения»;
-   ConnectionManager;
-   ConnectionStorage;
-   ConnectionJsonParser;
-   ConnectionEditForm;
-   добавление подключения;
-   редактирование подключения;
-   удаление подключения;
-   проверка корректности JSON;
-   контекстное меню;
-   отображение списка подключений в таблице.

CRUD функционал можно считать завершенным.

------------------------------------------------------------------------

# Следующие этапы разработки

## Этап 1. Завершить хранение подключений (Высокий приоритет)

### Цель

Обеспечить сохранение списка подключений между запусками приложения.

### Требования

-   проверить работу ConnectionStorage.Save();
-   проверить работу ConnectionStorage.Load();
-   определить место хранения файла `connections.json`;
-   если файла нет --- автоматически создать пустой;
-   при запуске приложения автоматически загружать подключения;
-   после любого изменения списка (добавление, удаление, редактирование)
    автоматически сохранять изменения;
-   после повторного запуска приложения список должен полностью
    восстанавливаться.

**Результат:** пользователь не должен повторно создавать подключения
после каждого запуска.

------------------------------------------------------------------------

## Этап 2. Реализовать систему валидаторов

Необходимо реализовать:

-   RequiredFieldsValidator
-   DuplicateTagValidator
-   TunValidator
-   SystemProxyValidator

Все проверки должны выполняться через ConnectionManager при:

-   Add()
-   Update()

UI не должен содержать бизнес-логики проверки.

------------------------------------------------------------------------

## Этап 3. Улучшить отображение таблицы

Сейчас отображается практически сырой JSON.

Необходимо сформировать компактное описание подключения.

Например:

### SOCKS

``` text
listen:127.0.0.1
port:1080
udp:true
```

### TUN

``` text
address:172.19.0.1/30
auto_route:true
strict_route:true
```

Отображение должно строиться автоматически на основании JSON.

------------------------------------------------------------------------

## Этап 4. Реализовать выбор активных подключений

Необходимо добавить возможность выбирать подключения, которые будут
использоваться при запуске.

Требования:

-   использовать чекбокс в таблице;
-   пользователь может выбрать несколько подключений;
-   одновременно может быть выбран только один TUN;
-   при выборе второго TUN необходимо вывести ошибку либо автоматически
    снять предыдущий выбор.

Информация о выбранных подключениях должна сохраняться.

------------------------------------------------------------------------

## Этап 5. Реализовать ConnectionBuilder

Создать новый сервис.

### Назначение

На основании выбранных подключений формировать секцию `inbounds`.

### Алгоритм

1.  считать `config.json`;
2.  удалить существующий массив `inbounds`;
3.  сформировать новый массив;
4.  вставить выбранные подключения;
5.  вернуть готовый JsonDocument либо сохранить временный config.

ConnectionBuilder не должен заниматься запуском sing-box.

------------------------------------------------------------------------

## Этап 6. Интеграция с запуском

Перед запуском необходимо:

1.  проверить, что выбрано хотя бы одно подключение;
2.  считать `config.json`;
3.  вызвать ConnectionBuilder;
4.  получить новый config;
5.  сохранить временный config;
6.  запускать sing-box уже с ним.

Если подключение не выбрано --- запуск запрещается.

------------------------------------------------------------------------

## Этап 7. Дополнительные улучшения

После завершения основной логики можно реализовать:

-   улучшенное отображение информации;
-   иконки типов подключений;
-   сортировку;
-   поиск;
-   экспорт/импорт подключений;
-   дополнительные валидаторы.

Эти задачи не являются приоритетными.
